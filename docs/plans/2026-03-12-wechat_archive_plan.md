# 独立产品人公众号历史文章归档计划

## 1. 项目目标

把公众号“独立产品人”的全部历史文章，按你当前仓库规范完整归档到 GitHub 仓库中，最终只保留两类核心资产：

- 原版 Markdown：`YYYY/MM/DD/article.md`
- 原始配图：`YYYY/MM/DD/images/*`

本次范围**不处理**以下内容：

- `wechat/article.html`
- 小红书图文分页
- 其他平台分发格式
- 评论、阅读量、转发量等扩展数据

最终交付目标是：博客站点后续只需要读取仓库中的 Markdown 和图片，就能稳定渲染历史文章。

---

## 2. 当前结论

### 2.1 仓库规范已经明确

当前仓库已经形成一套很清晰的落库标准：

- 文章目录：`YYYY/MM/DD/`
- 正文文件名固定：`article.md`
- 图片目录固定：`images/`
- 根目录索引文件：`metadata.json`

当前 `article.md` 的风格也已经稳定：

- 第一行直接是一级标题
- 不使用 YAML front matter
- 正文使用纯 Markdown
- 图片使用相对路径
- 文末可附原文来源说明

### 2.2 推荐主方案

主方案采用两层结构：

1. 使用 `wechat-article-exporter` 负责“历史文章发现 + 批量导出”
2. 自建一层“仓库标准化脚本”，负责把导出结果整理成你仓库需要的目录、图片、元数据格式

这是当前最稳、最省时间、最适合长期维护的方案。

### 2.3 推荐备方案

如果主方案无法完整覆盖全部历史文章，则使用半自动补洞链路：

1. 用 `mp-vx-insight` 或其他浏览器辅助方式拿到文章链接
2. 用 `wechat-to-markdown` 或自建转换脚本处理单篇文章
3. 对缺失文章执行人工复核后落库

---

## 3. 方案判断依据

### 3.1 最适合作为主引擎的开源项目

- `wechat-article/wechat-article-exporter`
  适合做全量发现、批量下载、Markdown 导出、图片下载
- `LuckyRyan-web/wechat-to-markdown`
  适合做单篇 URL 转 Markdown 的补充能力
- `pudongping/mp-vx-insight`
  适合做人工补洞，不适合做主链路

### 3.2 不建议作为主路线的项目

- `wechatsync/Wechatsync`
  更偏分发，不是历史归档
- `wearticle / Project WAN`
  更偏 HTML 标准化，不是 Markdown 资产沉淀

### 3.3 核心风险判断

最大的风险不是 Markdown 转换，而是“能不能稳定找全历史文章”。

根据现有文档与开源方案，当前公开可行路径大多依赖网页侧能力，而不是稳定、公开、官方的“历史文章一键导出 API”。这意味着：

- 可用，但需要留出人工兜底
- 工具失效时需要可替换方案
- 不能把长期资产完全依赖在第三方公共站点上

## 4. 总体架构

归档链路拆成五层，避免抓取逻辑和仓库标准耦合在一起。

### 第一层：文章发现层

目标：拿到“独立产品人”全部历史文章的文章清单。

输出建议字段：

- `title`
- `publish_date`
- `article_url`
- `cover_url`
- `author`
- `digest`
- `source_channel`

优先使用：

- `wechat-article-exporter`

兜底方式：

- 浏览器辅助插件
- 人工补录 URL
- 手工建立缺失清单

### 第二层：正文抓取层

目标：把每篇文章的正文内容和正文内图片链接提取出来。

输出建议字段：

- `title`
- `content_html`
- `content_markdown`
- `image_urls`
- `source_url`

原则：

- 不直接依赖导出后的 Markdown 为最终结果
- 保留中间结构，便于二次标准化

### 第三层：资产本地化层

目标：把封面图和正文图片全部下载到仓库本地。

规则：

- 每篇文章的所有图片必须本地化
- Markdown 中不得保留微信 CDN 临时地址作为最终链接
- 图片命名统一改成英文语义化或稳定编号

建议命名：

- `cover.jpg`
- `image-01.jpg`
- `image-02.png`

### 第四层：仓库标准化层

目标：把抓取结果转成仓库约定的最终结构。

每篇文章最终产物：

- `YYYY/MM/DD/article.md`
- `YYYY/MM/DD/images/*`

同时维护：

- `metadata.json`

统一规则：

- 目录日期使用文章发布日期
- `article.md` 第一行写标题
- 图片路径改写为相对路径
- 缺失图片要显式报错，不允许静默跳过
- `metadata.json.title` 与文章标题一致

### 第五层：校验与补洞层

目标：确保“归档完整”而不是“批量写入一堆半成品”。

至少做四类校验：

- 文章总数校验：导入数是否与发现数一致
- 图片校验：Markdown 中引用的图片是否全部存在
- 元数据校验：`metadata.json` 是否与目录结构一致
- 内容抽查：标题、日期、首图、正文段落是否明显缺失

对异常文章生成一份待处理清单，而不是直接视为完成。

## 5. 执行计划

以下计划按“先验证、再批量、再补洞”的顺序推进。

### 阶段 A：环境准备

目标：搭好后续归档使用的稳定工作环境。

任务：

1. 在本地私有环境部署 `wechat-article-exporter`
2. 验证是否可以搜索到公众号“独立产品人”
3. 验证是否可以列出历史文章
4. 确认导出 `md`、图片下载、文章链接导出是否可用

阶段完成标准：

- 可以稳定拿到一篇测试文章的导出结果

### 阶段 B：小样本打通

目标：先拿 3 到 5 篇文章做完整流程验证。

任务：

1. 选取几篇不同时间的文章做样本
2. 导出原始 Markdown 与图片
3. 编写标准化脚本
4. 落到仓库目标结构
5. 更新 `metadata.json`
6. 用博客侧预期消费方式验证是否可读

阶段完成标准：

- 3 到 5 篇文章能稳定落到仓库规范中
- 图片都能本地访问
- `metadata.json` 可正确索引这些文章

### 阶段 C：全量历史文章发现

目标：拿到“全量文章清单”，建立真正确认口径。

任务：

1. 批量导出文章列表
2. 建立文章主清单
3. 与仓库现有文章做去重比对
4. 产出“待归档文章列表”

阶段完成标准：

- 明确已有多少篇、缺多少篇、无法直接获取多少篇

### 阶段 D：全量批量归档

目标：把缺失文章批量写入仓库。

任务：

1. 按文章批量抓取正文与图片
2. 执行标准化脚本
3. 生成目录与文章文件
4. 统一更新 `metadata.json`
5. 执行完整性校验

阶段完成标准：

- 绝大多数历史文章完成自动归档
- 所有自动成功文章都满足仓库规范

### 阶段 E：人工补洞

目标：处理主流程未覆盖的文章。

适用场景：

- 搜不到的文章
- 导出失败的文章
- 图片丢失的文章
- 特殊版式导致 Markdown 错乱的文章

任务：

1. 生成异常文章清单
2. 用单篇工具或浏览器辅助方式补采
3. 人工校对标题、日期、图片、正文结构
4. 重新落库并复核

阶段完成标准：

- 异常文章被明确分类为“已修复”或“暂不可恢复”

### 阶段 F：增量同步机制

目标：避免以后又重复做一次历史补档。

任务：

1. 设计新增文章导入流程
2. 新文章发布后优先落原版 Markdown
3. 自动更新 `metadata.json`
4. 为后续定时同步预留脚本入口

阶段完成标准：

- 新文章以后可以走同一条归档标准，不再依赖手工散落保存

## 6. 需要产出的文件与脚本

后续正式进入开发阶段时，建议新增以下内容：

- `scripts/`
  存放导入、标准化、校验脚本
- `data/`
  存放中间产物，如文章清单、异常清单、导出缓存
- `docs/plans/`
  存放计划与执行记录

建议脚本拆分为三类：

- `discover_articles.*`
  负责拿到文章清单
- `normalize_article.*`
  负责把单篇结果转成仓库格式
- `validate_archive.*`
  负责做完整性校验

## 7. 落库规范

### 7.1 单篇文章目录

```text
YYYY/MM/DD/
├── article.md
└── images/
    ├── cover.jpg
    ├── image-01.jpg
    └── image-02.png
```

### 7.2 article.md 规范

- 第一行固定 `# 文章标题`
- 使用纯 Markdown
- 图片全部使用本地相对路径
- 不混入公众号 HTML 排版代码
- 文末统一保留原文来源说明

### 7.3 metadata.json 规范

每篇至少维护：

- `id`
- `date`
- `title`
- `path`
- `cover`
- `platforms.blog.status`
- `platforms.blog.url`

建议继续保留现有 `wechat` 与 `xiaohongshu` 结构占位，避免未来消费端分叉。

## 8. 风险与应对

### 风险 1：公众号无法被搜索到

应对：

- 先用主工具验证搜索可用性
- 不行就切到半自动补录方案

### 风险 2：导出的 Markdown 不符合仓库规范

应对：

- 明确“导出结果 != 最终入库结果”
- 必须经过标准化脚本

### 风险 3：图片链接失效或丢失

应对：

- 归档时强制图片本地化
- 校验 Markdown 中所有图片是否存在

### 风险 4：批量导入后 metadata 错乱

应对：

- 脚本统一生成或更新 `metadata.json`
- 导入后执行一致性校验

### 风险 5：部分老文章版式复杂

应对：

- 为异常文章保留人工修复流程
- 不把边角个例阻塞整个批量流程

---

## 9. 推荐执行顺序

建议按以下顺序推进，不建议直接一口气全量抓取：

1. 验证“独立产品人”能否被主工具发现
2. 选 3 到 5 篇文章跑通样本链路
3. 确认标准化后的目录和元数据没问题
4. 再做全量历史文章导入
5. 最后处理异常和补洞

---

## 10. 下一步建议

下一阶段不应该继续泛调研，而应该进入“样本验证”。

最合理的下一步是：

1. 部署并验证 `wechat-article-exporter`
2. 抓 3 到 5 篇“独立产品人”文章
3. 设计并实现标准化落库脚本
4. 验证导入结果是否满足博客读取要求

如果样本阶段跑通，后面的全量导入基本就是工程化执行问题，不再是方案不确定问题。
